Primary Science Explorer
Product Specification for Codex

Interactive web application for primary pupils aged 4–11, covering enquiry-led virtual investigations across EYFS/KS1/KS2.
Source basis: England science national curriculum (KS1 and KS2), PSTT enquiry skills progression, and the project investigation catalogue. This specification assumes working scientifically is embedded within substantive science content, not presented as a separate mini-game.
Prepared for: Codex implementation planning
Audience: product engineer / full-stack developer / UX builder
Purpose: turn the research and curriculum mapping into a phased, buildable application specification


1. Product vision
Primary Science Explorer is a browser-based science enquiry platform in which pupils carry out age-appropriate virtual investigations linked directly to the England primary science curriculum. The app is designed to help pupils move through the enquiry process from asking questions to evaluating findings, with the level of independence, recording and interpretation carefully adapted for each phase of schooling.
The product must cover all curriculum areas for science in Years 1–6, with at least one interactive investigation per curriculum area per year group, and should also include a Reception-ready on-ramp aligned to early observation, noticing change and talking about the natural world.
1.1 Core product principles
• Enquiry is embedded in content. Pupils learn to work scientifically through plants, animals, materials, light, electricity and other science areas, rather than in a detached generic skill game.
• The app is guided, not unguided. Pupils make choices, run tests and interpret outcomes, but the interface scaffolds each stage of the enquiry cycle so poor habits are not silently reinforced.
• The age range matters. A Year 1 investigation should privilege concrete observation, simple sorting and straightforward recording; a Year 6 investigation should support planning, variable control, repeat readings, graphing, explanation and evaluation.
• Every investigation should feel like doing science. Pupils must change something, observe something, record something, and make sense of evidence.
• The app supports classroom science rather than replacing it. It should be framed as preparation, rehearsal, consolidation, catch-up, accessibility support and curriculum coverage when real-world practical work is limited.
1.2 Target users
• Primary pupils aged 4–11, using pupil mode.
• Teachers, using teacher mode for assignment, projection, review and export.
• Developers/content authors, using an investigation definition system that allows new simulations to be added consistently.
2. Curriculum and pedagogy requirements
The national curriculum requires working scientifically to be taught through programme of study content, with KS1 pupils asking simple questions, observing closely, performing simple tests, identifying/classifying and gathering/recording simple data; LKS2 pupils then begin setting up simple enquiries, measuring more systematically and presenting data in tables, bar charts and labelled diagrams; UKS2 pupils plan enquiries, consider variables, take repeat readings, use more complex data representations and evaluate trust in results.
The PSTT enquiry skills guidance frames enquiry as a sequence involving asking questions, making predictions, setting up tests, observing and measuring, recording data, interpreting and communicating results, and evaluating. It also emphasises that teaching may focus on one enquiry skill while still using the others within a practical lesson.
The project investigation catalogue already maps one anchor simulation to every curriculum area in each year group and recommends a guided enquiry UX model, using enquiry types such as observing over time, pattern seeking, identifying/classifying, comparative/fair testing and research using secondary sources as a useful navigation taxonomy.

Product scope
3.1 In scope
• A web app that runs on common school devices, especially Chromebooks and desktop/laptop browsers.
• Pupil mode with guided investigations, recording tools, autosave and exports.
• Teacher mode with launch, assign, preview, review and export functions.
• A reusable simulation framework so all investigations share a common enquiry shell.
• Curriculum mapping and metadata so content can be filtered by year group, topic, enquiry type and working scientifically objective.
• Accessibility features and inclusive interaction models.
3.2 Out of scope for first release
• Trying to simulate every possible practical detail with full scientific realism.
• Open internet research inside the pupil experience.
• Native mobile apps.
• High-stakes summative assessment claims.
• Complex multi-user real-time collaboration.
4. User experience model
4.1 Core app structure
• Home dashboard: choose year group, curriculum area, investigation or assigned task.
• Investigation shell: the universal workflow wrapper around each simulation.
• Notebook: stores prediction, planning notes, results, graphs, diagrams and conclusion/evaluation responses.
• Teacher area: class view, assignment tools, exemplar mode, exports and curriculum coverage reporting.
4.2 The universal investigation shell
Every simulation should sit inside one consistent shell so pupils do not have to relearn the interface. The shell should include: question, prediction, plan, simulation workspace, record panel, analysis panel, conclusion prompt, evaluation prompt, and a progress indicator.

• Reception and KS1: icon-rich navigation, audio support, very short text, large controls, minimal simultaneous options, strong modelling, simple recording formats.
• Years 3 and 4: introduce tables, measured values, simple graphing, explicit comparative testing and branching keys.
• Years 5 and 6: add variable panels, repeat trials, graph choice, evidence prompts, misconception checks and evaluation of trust in results.
4.4 Teacher experience
• Preview any investigation in teacher mode with all steps unlocked.
• Assign an investigation to a year group/class or launch as a whole-class projected activity.
• View completed pupil notebooks and exports.
• See which curriculum areas and enquiry types have been covered.
• Optionally simplify, hide or prefill sections for support.
5. Functional requirements
5.1 Content and metadata
• Each investigation must be defined by structured metadata: year group, curriculum area, topic keywords, enquiry type, working scientifically objectives, age band, complexity, safety note, and linked prompt set.
• The system must support future authoring of new investigations without rebuilding the whole app.
• Each investigation must include teacher notes and pupil-facing instructions.
5.2 Simulation engine requirements
• Support common simulation types: time-based growth/change, drag-and-drop classification, simple physics comparisons, circuit logic, ray tracing, charts and graph outputs, simple ecosystems and agent/probabilistic models.
• Allow repeat trials with controlled randomness where educationally useful.
• Expose measurable outputs to the record panel automatically.
• Allow teacher-defined presets so the same investigation can run at different difficulty levels.
5.3 Recording and notebook requirements
• Notebook autosaves at every step.
• Pupils can revisit earlier steps without losing later work.
• Investigation output may include drawings, labels, tables, bar charts, line graphs, scatter graphs, branching keys, short paragraphs and slide-style report cards, depending on phase.
• Exports should support at least PDF and print-friendly view for teacher use.
5.4 Assessment and reporting requirements
• Capture process data, not only final answers: question chosen, prediction made, variable choices, number of trials, recording choices, conclusion text and evaluation text.
• Provide teacher-readable summaries showing whether pupils completed each enquiry stage.
• Support simple evidence statements such as completed / attempted / needs review rather than misleading automated grading claims.
5.5 Accessibility and inclusion requirements
• Keyboard-operable alternatives for drag interactions.
• Clear focus states and logical tab order.
• Text alternatives for essential audio information.
• Narration or read-aloud option for younger pupils and emerging readers.
• Colour should never be the only carrier of meaning.
• Volume-limited sound tools and a mute-safe default for sound investigations.

Technical architecture recommendation
Codex should build the application around a shared investigation framework rather than as a collection of isolated pages. The most important architectural decision is to separate the enquiry shell, the simulation components and the content metadata.
7.1 Suggested stack
• Frontend: React or Next.js with TypeScript.
• Rendering: Canvas/SVG/WebGL selectively depending on the investigation; use the simplest adequate option.
• State: a predictable client store for current investigation state plus persisted notebook data.
• Charts: a reliable accessible chart library for bar/line/scatter outputs.
• Backend: lightweight API for authentication (if used), classes/assignments, saved notebooks and reporting.
• Database: structured relational or document storage keyed by investigation metadata and user/session records.

• Run content QA against curriculum coverage, age appropriateness and misconception handling.

Content design requirements for every investigation
• Pupil title and concise challenge statement.
• Teacher note with curriculum link and intended learning.
• Named enquiry type and working scientifically objectives.
• Age-appropriate question prompt.
• Prediction prompt.
• Plan step with only the level of complexity suitable for that phase.
• Simulation controls and variable labels.
• Auto-captured or pupil-entered recording area.
• Interpretation prompt with evidence requirement.
• Evaluation prompt.
• Safety note linking the virtual task to sensible real-world science behaviour.
11. Misconceptions and pedagogical safeguards
• Do not let pupils progress through an investigation without recording some evidence.
• Do not present random outcomes as mistakes; where randomness exists, teach repetition and patterns across runs.
• Use misconception checks in content where there are common errors, for example: dark is the absence of light; bulbs do not 'use up' electricity in a simple explanatory sense; animals do not choose to adapt because they want to.
• Avoid over-precise scientific claims when the curriculum only requires simplified models.
• Show when a simulation is a model. Older pupils should be told that a model helps explain but is not the thing itself.
12. QA and acceptance criteria
12.1 Global acceptance criteria
• Every year group and curriculum area required by the source catalogue is represented by at least one working investigation.
• Every investigation can be completed from question to evaluation within the shared enquiry shell.
• Age-appropriate recording modes are enforced.
• Teacher mode can preview and export outputs.
• Autosave and resume work reliably.
• Accessibility checks pass for core navigation and essential tasks.
12.2 Per-investigation acceptance checklist
• Loads correctly and presents the right metadata.
• Question, prediction, testing, recording, interpretation and evaluation steps all function.
• Simulation outputs can be recorded and revisited.
• Pupil can complete without encountering dead ends.
• Teacher can reset or demo the investigation.
• Copy is accurate to the intended curriculum content.
13. Suggested build order of investigations
Codex should not implement the anchor investigations in curriculum order. The better order is by reusability and dependency.
• Wave 1: Habitat Match and Explain; Material Properties Test Bench; Senses Sorting; Daylight Tracker.

16. Final implementation brief for Codex
Build Primary Science Explorer as a metadata-driven, enquiry-first web application. Start by creating the shared investigation shell, notebook and content schema. Then implement investigations in reusable families, beginning with KS1-friendly low-complexity tasks and expanding to LKS2 and UKS2 advanced models. Do not treat simulations as isolated mini-games: every investigation must move pupils through question, prediction, plan, test, record, interpret and evaluate, with the level of support adapted to age.
